# VPS CI/CD Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Deploy the exact tested `main` commit to the existing VPS through GitHub Actions, with serialized deployments, migration execution, local/public health checks, and automatic code rollback.

**Architecture:** Extend the existing `.github/workflows/ci.yml` with a production deploy job that depends on both current CI jobs. The job authenticates with a dedicated SSH key and pinned known-host entry, then invokes a repository-owned `scripts/deploy-vps.sh` on the VPS. The script owns locking, exact-SHA checkout, Docker Compose build/migration/restart, health polling, diagnostics, and rollback.

**Tech Stack:** GitHub Actions, OpenSSH, Bash, Git, Docker Compose v2, Flask-Migrate, curl, pytest.

- [ ] **Step 1: Create the deployment script**

```bash
#!/usr/bin/env bash
set -Eeuo pipefail

TARGET_SHA="${1:?Usage: deploy-vps.sh <git-sha>}"
APP_DIR="${APP_DIR:-$(pwd)}"
SERVICE="${DEPLOY_SERVICE:-web}"
HEALTH_URL="${DEPLOY_HEALTH_URL:-http://127.0.0.1:8000/health/}"
PUBLIC_HEALTH_URL="${DEPLOY_PUBLIC_HEALTH_URL:-}"
HEALTH_ATTEMPTS="${DEPLOY_HEALTH_ATTEMPTS:-30}"
HEALTH_INTERVAL="${DEPLOY_HEALTH_INTERVAL:-2}"
LOCK_FILE="${DEPLOY_LOCK_FILE:-/tmp/ozzyl-tools-deploy.lock}"
PREVIOUS_SHA=""

log() {
  printf '[deploy] %s\n' "$*"
}

wait_for_health() {
  local url="$1"
  local label="$2"
  local attempt
  for ((attempt = 1; attempt <= HEALTH_ATTEMPTS; attempt += 1)); do
    if curl --fail --silent --show-error --max-time 5 "$url" >/dev/null; then
      log "${label} health check passed on attempt ${attempt}."
      return 0
    fi
    sleep "$HEALTH_INTERVAL"
  done
  log "${label} health check failed after ${HEALTH_ATTEMPTS} attempts."
  return 1
}

deploy_commit() {
  local sha="$1"
  log "Checking out ${sha}."
  git checkout --detach "$sha"

  log "Building ${SERVICE}."
  docker compose build --pull "$SERVICE"

  log "Applying database migrations."
  docker compose run --rm "$SERVICE" flask --app run:app db upgrade

  log "Recreating ${SERVICE}."
  docker compose up -d --no-deps --force-recreate "$SERVICE"

  wait_for_health "$HEALTH_URL" "Local"
  if [[ -n "$PUBLIC_HEALTH_URL" ]]; then
    wait_for_health "$PUBLIC_HEALTH_URL" "Public"
  fi
}

rollback() {
  if [[ -z "$PREVIOUS_SHA" ]]; then
    log "No rollback SHA is available."
    return 1
  fi

  log "Rolling back to ${PREVIOUS_SHA}."
  git checkout --detach "$PREVIOUS_SHA"
  docker compose build "$SERVICE"
  docker compose up -d --no-deps --force-recreate "$SERVICE"
  wait_for_health "$HEALTH_URL" "Rollback local"
  log "Rollback health check passed."
}

on_error() {
  local exit_code=$?
  trap - ERR
  log "Deployment failed with exit code ${exit_code}."
  docker compose ps || true
  docker compose logs --tail=120 "$SERVICE" || true
  rollback || log "Rollback failed; manual intervention is required."
  exit "$exit_code"
}

exec 9>"$LOCK_FILE"
if ! flock -n 9; then
  log "Another deployment is already running."
  exit 75
fi

cd "$APP_DIR"

if [[ "$(git rev-parse --is-inside-work-tree)" != "true" ]]; then
  log "APP_DIR is not a Git repository: ${APP_DIR}"
  exit 2
fi

if ! git diff --quiet || ! git diff --cached --quiet; then
  log "Tracked local changes detected; refusing to overwrite the VPS checkout."
  exit 3
fi

PREVIOUS_SHA="$(git rev-parse HEAD)"
trap on_error ERR

log "Fetching origin/main."
git fetch --prune origin main

git cat-file -e "${TARGET_SHA}^{commit}"
deploy_commit "$TARGET_SHA"

mkdir -p instance
printf '%s\n' "$TARGET_SHA" > instance/deployed-sha
trap - ERR
log "Deployment completed for ${TARGET_SHA}."
```

```yaml
  deploy:
    needs: [test, docker]
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    timeout-minutes: 20
    environment: production
    concurrency:
      group: production-deploy
      cancel-in-progress: true
    env:
      VPS_HOST: ${{ secrets.VPS_HOST }}
      VPS_USER: ${{ secrets.VPS_USER }}
      VPS_PORT: ${{ secrets.VPS_PORT }}
      VPS_SSH_KEY: ${{ secrets.VPS_SSH_KEY }}
      VPS_KNOWN_HOSTS: ${{ secrets.VPS_KNOWN_HOSTS }}
      VPS_APP_DIR: ${{ secrets.VPS_APP_DIR }}
      PRODUCTION_URL: ${{ secrets.PRODUCTION_URL }}
    steps:
      - name: Validate deployment secrets
        run: |
          test -n "$VPS_HOST"
          test -n "$VPS_USER"
          test -n "$VPS_PORT"
          test -n "$VPS_SSH_KEY"
          test -n "$VPS_KNOWN_HOSTS"
          test -n "$VPS_APP_DIR"
          test -n "$PRODUCTION_URL"

      - name: Configure SSH
        run: |
          install -m 700 -d "$HOME/.ssh"
          printf '%s\n' "$VPS_SSH_KEY" > "$HOME/.ssh/deploy_key"
          chmod 600 "$HOME/.ssh/deploy_key"
          printf '%s\n' "$VPS_KNOWN_HOSTS" > "$HOME/.ssh/known_hosts"
          chmod 600 "$HOME/.ssh/known_hosts"

      - name: Deploy tested commit to VPS
        env:
          GITHUB_SHA: ${{ github.sha }}
        run: |
          ssh \
            -i "$HOME/.ssh/deploy_key" \
            -p "$VPS_PORT" \
            -o BatchMode=yes \
            -o StrictHostKeyChecking=yes \
            "$VPS_USER@$VPS_HOST" \
            "cd '$VPS_APP_DIR' && DEPLOY_PUBLIC_HEALTH_URL='${PRODUCTION_URL%/}/health/' ./scripts/deploy-vps.sh '$GITHUB_SHA'"

      - name: Verify public production health
        run: |
          curl \
            --fail \
            --show-error \
            --silent \
            --retry 10 \
            --retry-delay 3 \
            --retry-all-errors \
            "${PRODUCTION_URL%/}/health/"
```

## VPS deployment

The `main` branch deploys automatically only after Python, JavaScript, migration, audit, and Docker checks pass.

### VPS prerequisites

Install Git, Docker Engine, Docker Compose v2, `curl`, and `flock`. Keep the repository checked out at the directory stored in `VPS_APP_DIR`, and keep the production `.env` file on the VPS. The deployment script never commits or replaces `.env` or `instance/`.

Create a dedicated SSH key for GitHub Actions and add only its public key to the deployment user's `~/.ssh/authorized_keys`.

Capture the host key from a trusted machine:

```bash
ssh-keyscan -H -p 22 103.72.65.220
```

Verify the fingerprint through the VPS provider console before saving the complete output as `VPS_KNOWN_HOSTS`.

### GitHub production secrets

Configure the `production` environment with:

- `VPS_HOST` — `103.72.65.220`
- `VPS_USER` — `root` until a restricted deploy user is introduced
- `VPS_PORT` — the actual SSH port
- `VPS_SSH_KEY` — the dedicated private deployment key
- `VPS_KNOWN_HOSTS` — verified `known_hosts` line
- `VPS_APP_DIR` — absolute repository path on the VPS
- `PRODUCTION_URL` — public HTTPS origin without a trailing slash

### Release behavior

The VPS checks out the exact successful GitHub SHA, builds the `web` service, runs `flask --app run:app db upgrade`, recreates the container, and checks both `http://127.0.0.1:8000/health/` and the public Cloudflare-backed `/health/` endpoint. GitHub performs a second external public health check after SSH completes.

If build, migration, startup, local health verification, or public health verification fails, the script checks out and restarts the previous commit. Database downgrades are intentionally not automatic, so migrations should remain backward compatible for at least one release.
```
